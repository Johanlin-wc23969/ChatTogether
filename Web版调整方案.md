# 匿名语音辩论聊天室 Web 优先调整方案

## 1. 为什么优先做 Web 版

从 MVP 和朋友内测角度看，Web 版比 iOS 原生 App 更适合作为第一阶段产品形态。

主要原因：

- 不需要 Apple Developer Program 账号。
- 不需要 App Store 审核。
- 不需要 TestFlight 名额和审核流程。
- 朋友内测只需要打开链接。
- 可以快速迭代房间规则、UI 和玩法。
- WebRTC 在现代浏览器中已经能支持实时语音。
- 后续如果玩法验证成功，再封装成 iOS App 或 PWA。

因此建议项目路线调整为：

```text
第一阶段：Web MVP
第二阶段：PWA 优化，支持添加到主屏幕
第三阶段：根据数据决定是否开发 iOS 原生 App
```

## 2. 产品形态调整

### 2.1 Web MVP

用户通过浏览器访问网页，无需下载安装。

核心入口：

- 创建房间。
- 通过链接加入房间。
- 输入或自动生成匿名会话身份。
- 授权麦克风。
- 进入匿名语音辩论房。

推荐 MVP 内测方式：

```text
房主创建房间 -> 复制邀请链接 -> 发给朋友 -> 朋友浏览器打开 -> 加入房间
```

### 2.2 暂不做复杂房间列表

MVP 阶段建议先不做公开房间列表，原因是：

- 更利于控制内测范围。
- 避免陌生人内容审核压力过早出现。
- 邀请链接能更快验证核心玩法。

后续再加：

- 公开房间大厅。
- 推荐房间。
- 主题匹配。
- 随机加入。

## 3. Web 推荐技术栈

### 3.1 前端

推荐：

```text
框架：Next.js 或 Vite + React
语言：TypeScript
UI：Tailwind CSS
状态管理：Zustand 或 React Context
实时状态：WebSocket
实时语音：LiveKit Web SDK 或 Agora Web SDK
部署：Vercel / Cloudflare Pages
```

如果目标是最快做出 MVP：

```text
Vite + React + TypeScript + Tailwind CSS
```

如果希望后续有 SEO、服务端渲染、登录系统和更完整产品结构：

```text
Next.js + TypeScript + Tailwind CSS
```

### 3.2 后端

仍然推荐 Go：

```text
语言：Go
HTTP API：Gin / Echo
WebSocket：nhooyr.io/websocket 或 gorilla/websocket
数据库：PostgreSQL
缓存：Redis
部署：Fly.io / Render / Railway / AWS / GCP
```

### 3.3 实时语音

Web 版建议优先考虑：

```text
LiveKit：适合 WebRTC、自托管、长期掌控基础设施
Agora：适合快速上线、移动端和 Web 端成熟、变声能力较完善
```

如果 MVP 重点是“最快让朋友听到彼此声音”，Agora 会更省事。

如果你希望未来控制音频房间、服务器和成本，LiveKit 更适合长期路线。

## 4. Web 架构方案

```text
Browser
  |-- HTTPS REST API：创建房间、加入房间、获取 token
  |-- WebSocket：房间状态、队列、倒计时、发言权
  |-- WebRTC SDK：语音收听与发言

Frontend Hosting
  |-- Vercel / Cloudflare Pages

Go Backend
  |-- Auth Service
  |-- Room Service
  |-- Realtime Gateway
  |-- Topic Service
  |-- Voice Token Service

Redis
  |-- 房间实时状态
  |-- 发言队列
  |-- 冷却状态
  |-- WebSocket Pub/Sub

PostgreSQL
  |-- 用户
  |-- 房间
  |-- 辩题
  |-- 举报

LiveKit / Agora
  |-- 语音房间
  |-- 麦克风发布权限
  |-- 变声或音效处理
```

## 5. Web 端页面设计

### 5.1 首页

功能：

- 创建房间。
- 输入邀请链接或房间码加入。
- 选择主题分类。
- 选择房间人数：2-6 人。

### 5.2 创建房间页

功能：

- 选择主题分类。
- 选择房间人数。
- 创建房间。
- 创建成功后显示邀请链接。

### 5.3 加入房间页

功能：

- 展示房间辩题。
- 展示人数和是否已开始。
- 请求麦克风权限。
- 点击加入。

注意：

- 浏览器必须在 HTTPS 或 localhost 下才能稳定使用麦克风权限。
- iPhone Safari 对 WebRTC 支持可用，但需要重点测试。

### 5.4 房间页

功能：

- 顶部展示辩题。
- 展示正方和反方阵营。
- 展示匿名卡通身份。
- 展示当前发言者。
- 展示 60 秒倒计时。
- 展示申请发言按钮。
- 展示 30 秒冷却。
- 展示发言队列。
- 提供复制邀请链接。

按钮状态：

- 等待开始。
- 申请发言。
- 排队中。
- 冷却中。
- 当前发言中，显示结束发言。
- 等待对方阵营。

## 6. 匿名身份与变声在 Web 端的处理

### 6.1 匿名身份

后端继续负责分配：

- 卡通角色。
- 阵营。
- 变声预设。

示例：

```json
{
  "persona": {
    "id": "bear",
    "name": "熊",
    "avatarUrl": "/personas/bear.png",
    "voicePreset": "deep_bear"
  },
  "side": "pro"
}
```

### 6.2 Web 变声方案

Web 上有三种方案：

#### 方案 A：使用 SDK 自带变声

如果使用 Agora，优先使用其 Web SDK 支持的音效或变声能力。

优点：

- 实现快。
- 稳定性较好。
- 不需要自己写复杂音频处理。

缺点：

- 效果和参数受 SDK 限制。
- 成本依赖第三方平台。

#### 方案 B：Web Audio API 本地处理

浏览器通过 Web Audio API 对麦克风输入进行 pitch、filter、distortion 等处理。

优点：

- 不依赖第三方变声能力。
- 可控性更强。

缺点：

- 实现复杂。
- 不同浏览器效果不一致。
- iOS Safari 兼容性要重点验证。

#### 方案 C：第一版只做轻量音色差异

MVP 可以先做轻量变声：

- 降低或提高 pitch。
- 增加轻微滤波。
- 保证声音可懂。

建议：

```text
第一版优先 SDK 变声。
如果 SDK 不满足，再补 Web Audio API。
```

## 7. Web MVP 开发拆分

## 7.1 前端开发

### 第一阶段：静态产品体验

- 搭建 React / Next.js 项目。
- 实现首页。
- 实现创建房间页。
- 实现房间页静态 UI。
- 实现匿名角色展示。
- 实现正反方布局。

### 第二阶段：接入后端 API

- 匿名会话创建。
- 创建房间。
- 通过邀请链接加入房间。
- 获取房间状态。
- 复制邀请链接。

### 第三阶段：接入 WebSocket

- 连接房间 WebSocket。
- 监听房间状态变化。
- 更新成员列表。
- 更新发言队列。
- 更新当前发言者。
- 展示倒计时和冷却。

### 第四阶段：接入语音

- 请求麦克风权限。
- 加入 LiveKit / Agora 房间。
- 默认关闭麦克风。
- 当前用户获得发言权时启用麦克风。
- 发言结束后关闭麦克风。
- 应用变声预设。

### 前端验收标准

- 用户可以通过链接加入房间。
- 2-6 人能在同一房间看到实时状态。
- 当前发言者 UI 明确。
- 申请发言按钮状态正确。
- 非当前发言者无法发出声音。
- 移动端浏览器布局可用。

## 7.2 后端开发

Web 版后端与原方案基本一致，但可以简化登录和发布。

### 第一阶段：匿名会话与房间

- 创建匿名 session。
- 创建房间。
- 生成邀请链接。
- 加入房间。
- 分配阵营。
- 分配匿名角色。
- 返回辩题。

### 第二阶段：实时状态

- 建立 WebSocket gateway。
- 维护房间内连接。
- 广播成员加入和退出。
- 广播完整 room_state。

### 第三阶段：发言规则

- 实现申请发言。
- 实现正反方队列。
- 实现 30 秒冷却。
- 实现 60 秒发言上限。
- 实现提前结束。
- 实现断线释放发言权。

### 第四阶段：语音 token

- 为每个房间创建语音房间 ID。
- 为用户签发 LiveKit / Agora token。
- 根据是否是当前发言者控制 publish 权限，MVP 可先由客户端控制。

### 后端验收标准

- 所有房间规则由后端判断。
- 客户端不能通过伪造状态获得发言权。
- 多人同时点击申请发言时队列稳定。
- 发言超时后自动切换阵营。
- 断线不会卡住房间。

## 7.3 实时语音开发

### 第一阶段：基础语音房间

- 用户加入语音房间。
- 所有人可以收听。
- 默认不发布麦克风。

### 第二阶段：发言权联动

- WebSocket 收到 speaker_started。
- 如果 speakerId 是当前用户，启用麦克风。
- 其他用户保持静音发布或不发布。
- speaker_ended 后关闭麦克风。

### 第三阶段：变声

- 根据 persona.voicePreset 初始化变声参数。
- 开麦前确认变声已应用。
- 变声失败时禁止开麦。

### 语音验收标准

- 同一时间只有一人能被听到。
- 发言者提前结束后立刻静音。
- 60 秒到期后自动静音。
- 刷新页面后可重连。
- iPhone Safari、Android Chrome、桌面 Chrome 至少完成基础测试。

## 8. 部署与内测方案

### 8.1 最快内测组合

```text
前端：Vercel
后端：Render / Railway / Fly.io
数据库：Supabase Postgres
Redis：Upstash Redis
语音：Agora 或 LiveKit Cloud
```

优点：

- 不需要自建服务器。
- 可以快速发链接给朋友。
- HTTPS 自动配置。
- 适合 MVP 验证。

### 8.2 内测流程

1. 部署 Web 站点。
2. 创建测试房间。
3. 复制邀请链接。
4. 发给 1-5 个朋友。
5. 收集以下反馈：
   - 是否愿意开口。
   - 匿名和变声是否降低压力。
   - 60 秒是否合适。
   - 30 秒冷却是否合适。
   - 正反轮流是否自然。
   - 是否需要文字弹幕或投票。

## 9. 对原 iOS 方案的调整结论

原 iOS 方案不废弃，但建议后置。

新的优先级：

```text
P0：Web MVP
P1：PWA 移动端体验优化
P2：完善内容审核和举报
P3：公开房间大厅和匹配
P4：iOS 原生 App
```

Web 版成功后，可以复用：

- Go 后端。
- Redis 房间状态。
- PostgreSQL 数据。
- 题库系统。
- 实时语音服务。
- 发言队列规则。

未来做 iOS App 时，主要新增的是原生客户端，不需要推翻后端架构。

## 10. 建议下一步

建议下一步直接做 Web MVP：

1. 确定前端框架：Vite React 或 Next.js。
2. 确定语音 SDK：Agora 或 LiveKit。
3. 先做无语音版本，验证房间状态机和发言队列。
4. 再接入语音和变声。
5. 部署到公网，用邀请链接内测。
